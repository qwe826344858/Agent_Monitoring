# 回归验证技能

## 适用场景

1. **阶段性迭代完成后** — 一批功能开发交付，需全面确认历史功能未被破坏
2. **用户主动要求** — "跑一遍回归"、"验证下历史功能"
3. **重大改动后** — 修改了核心模块、公共基础设施、协议层等高风险区域

## 前置条件

项目中必须存在 `docs/回归验证功能点清单.md`，内容包含：

1. **功能点总览表** — 所有历史功能点的编号、描述、归属阶段、是否有自动化测试
2. **功能点详细定义** — 每个功能点的验收标准
3. **自动化测试汇总** — 测试文件、用例数、执行命令
4. **端到端回归场景** — 场景列表、覆盖的功能点、每个场景的检查项
5. **回归记录表** — 历次回归的日期、结果、执行人

> 如果项目中尚未创建此清单，先读取项目代码和测试文件，生成初始版本。

## 执行流程

### 阶段 1：读取清单，确认范围

1. 读取 `docs/回归验证功能点清单.md`
2. 记录本次回归的触发原因
3. 检查本次迭代是否新增/修改了功能点：
   - **有新增** → 先更新清单（追加功能点编号、描述、验收标准、自动化覆盖状态），再执行验证
   - **无新增** → 直接进入阶段 2

### 阶段 2：全量自动化测试

读取清单中 §三 记录的测试命令并执行。常见模式：

```bash
# 后端测试（根据项目实际框架调整）
cd {backend_dir} && {test_command}

# 前端测试（根据项目实际框架调整）
cd {frontend_dir} && {test_command}

# 类型检查（如适用）
cd {frontend_dir} && {typecheck_command}
```

**通过标准**：全部 passed，0 failed。

**如果有失败**：
- 记录失败的测试名和错误信息
- 分析是"代码 bug"还是"测试本身过时"
- 不继续后续阶段，先修复后重新触发

### 阶段 3：端到端回归

读取清单中 §四 记录的端到端场景，按场景执行：

1. 如清单中记录了自动化端到端脚本路径 → 执行脚本
2. 如需要部署到远程/测试环境 → 按清单中记录的部署步骤执行
3. 收集每个场景的执行结果

### 阶段 4：逐项验证

对照清单中每个端到端场景的检查项，逐一核对：

- 每个检查项标记 ✅ 通过 / ❌ 不通过
- 不通过项记录具体问题描述

### 阶段 5：输出回归报告

**向用户输出简要总结**：

```
## 回归验证结果

- 触发原因：{原因}
- 自动化测试：{X} passed / {Y} failed
- 端到端回归：{M}/{N} 场景通过
- 不通过项：{列出具体场景和问题，无则写"无"}
- 结论：通过 ✅ / 不通过 ❌（需修复后重新回归）
```

**更新清单文档**：在 `docs/回归验证功能点清单.md` 的"回归记录"表追加一行：

```markdown
| {日期} | {触发原因} | {自动化结果} | {端到端结果} | {执行人} | {备注} |
```

### 阶段 6：（仅功能变更时）更新清单

当本次迭代涉及功能变更时，同步更新清单对应章节：

| 变更类型 | 更新位置 |
|---|---|
| 新增功能 | §一 追加行 + §二 追加详细定义 |
| 修改功能 | §二 更新对应验收标准 |
| 新增测试文件 | §三 更新用例数和文件列表 |
| 新增端到端场景 | §四 追加场景 + 检查项 |
| 删除功能 | §一 标记废弃 + §二 移除或标注 |

## 失败处理

| 情况 | 处理 |
|---|---|
| 自动化测试失败 | 停止回归，优先修复代码，修复后重新触发 |
| 端到端场景不通过 | 记录问题到回归记录备注，创建修复任务 |
| 测试/部署环境不可用 | 跳过对应阶段，仅执行可用部分，备注原因 |
| 清单文档不存在 | 先读取项目代码和测试，生成初始版清单 |

## 原则

1. **先验证，后修复** — 回归过程中不修改业务代码，完整跑完记录问题后再统一修复
2. **结果必须落文档** — 每次回归结果写入清单的回归记录表，不允许只在对话中报告
3. **清单是唯一真实来源** — 所有功能点、验收标准、测试命令以清单文档为准
4. **及时更新** — 每次需求变更必须同步更新清单，不允许功能点与代码脱节
