# Agent Teams 实时监控面板 — 设计方案

## 一、项目目标

可视化展示 Claude Code Agent Teams 中每个 Agent 的运行状态、任务进度和实时通信流，帮助用户直观理解多 Agent 协作过程。

## 二、系统架构

```
┌─────────────────────────────────────────────────────┐
│                    文件系统                           │
│  ~/.claude/teams/{team}/config.json     ← 成员信息   │
│  ~/.claude/teams/{team}/inboxes/*.json  ← 消息通信   │
│  ~/.claude/tasks/{team}/*.json          ← 任务状态   │
└──────────────┬──────────────────────────────────────┘
               │ fsnotify 监听文件变化
               ▼
┌──────────────────────────────┐
│      Go Backend (Gin)        │
│                              │
│  ┌────────────┐ ┌──────────┐ │
│  │ FileWatcher │ │ REST API │ │
│  │ (fsnotify)  │ │ /api/*   │ │
│  └─────┬──────┘ └──────────┘ │
│        │                     │
│  ┌─────▼──────┐              │

## 三、技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| 后端框架 | Go + Gin | 高性能，编译为单二进制 |
| 文件监听 | fsnotify | Go 原生，基于 inotify，低延迟 |
| 实时推送 | gorilla/websocket | 成熟稳定，双向通信 |
| 前端 | HTML + CSS + JS | 零构建依赖，`go:embed` 嵌入二进制 |
| 部署 | 单进程 | 一个可执行文件，零外部依赖 |

      "joinedAt": 1774863762841,
      "tmuxPaneId": "",
      "cwd": "/home/peiqi/...",
      "subscriptions": []
    }
  ]
}
```

### 4.2 消息通信 — `~/.claude/teams/{team}/inboxes/{agent-name}.json`

```json
[
  {
    "from": "team-lead",
    "text": "请开始处理任务 #1",
    "summary": "分配任务1",
    "timestamp": "2026-03-30T09:42:16.890Z",
    "read": false
  }
]
```

### 4.3 任务状态 — `~/.claude/tasks/{team}/*.json`

每个任务一个 JSON 文件，包含 ID、标题、状态、owner、依赖等。

辅助文件：
- `.lock` — flock 互斥锁，防止多 Agent 同时认领
- `.highwatermark` — 任务 ID 自增计数器

### 4.4 Token 消耗 — JSONL 会话记录

每个 Agent 的 token 消耗数据存储在 JSONL 会话记录文件中：

```
~/.claude/projects/{project-hash}/{session-id}.jsonl                    ← Team Lead
~/.claude/projects/{project-hash}/{session-id}/subagents/
  ├── agent-{id1}.jsonl                                                 ← 成员 A
  ├── agent-{id2}.jsonl                                                 ← 成员 B
  └── agent-acompact-{id3}.jsonl                                        ← 成员 C
```

其中 `{project-hash}` 是项目路径的转义形式（`/` 替换为 `-`），例如 `/home/peiqi/tmp/Agent_Monitoring` → `-home-peiqi-tmp-Agent_Monitoring`。

JSONL 每行是一个 JSON 对象，assistant 类型的消息包含 `usage` 字段：

```json
{
  "type": "message",
  "role": "assistant",
  "sessionId": "c38d08d9-...",
  "timestamp": "2026-03-31T10:00:00Z",
  "usage": {
    "input_tokens": 15000,
    "output_tokens": 2500,
    "cache_creation_input_tokens": 500,
    "cache_read_input_tokens": 1000
  }
}
```

Token 字段说明：

| 字段 | 说明 |
|------|------|
| `input_tokens` | 常规输入 token |
| `output_tokens` | 模型生成的输出 token |
| `cache_creation_input_tokens` | 写入 prompt cache 的 token |
| `cache_read_input_tokens` | 从 cache 读取的 token（成本为输入的 10%） |

#### 精确度问题

JSONL 是流式写入的，`input_tokens` 在流式响应开始时就已写入文件，此时值往往为 0 或占位值，**写入后不会回填更新**。因此：

- `input_tokens` 严重偏低（可能偏差 100x+）
- `output_tokens` 也偏低（偏差 10x+）
- **适用于**：相对趋势分析、成员间消耗比例对比、cache 命中率分析
- **不适用于**：精确计费

#### 团队 Token 汇总方式

```
团队总消耗 = Lead JSONL 的 usage 累计
           + 所有 subagents/*.jsonl 的 usage 累计
```

通过 `config.json` 中的 `leadSessionId` 关联到对应的 JSONL 目录。

#### 精确计费的补充方案

| 方式 | 精确度 | 粒度 | 获取方法 |
|------|--------|------|---------|
| JSONL `usage` 字段 | 不精确（趋势参考） | 每条消息 | 解析本地文件 |
| `/cost` 命令 | 精确 | 当前会话 | Bash 执行 `claude /cost` |
| Claude Code Analytics API | 精确 | 每日/每用户 | `POST /v1/organizations/usage_report/claude_code`（需 admin key） |

监控面板采用 JSONL 解析作为实时数据源，展示各成员的相对消耗比例和趋势。

#### 费用参考（按模型）

| 模型 | 输入 ($/M tokens) | 输出 ($/M tokens) |
|------|-------------------|-------------------|
| claude-opus-4-6 | $15 | $75 |
| claude-sonnet-4-6 | $3 | $15 |
| claude-haiku-4-5 | $0.80 | $4 |

Cache 读取 token 成本为输入 token 的 10%。

### 5.4 文件监听逻辑

```
fsnotify 监听:

~/.claude/teams/{team}/config.json
  → 变化时读取并对比成员列表 → 推送 member_update

~/.claude/teams/{team}/inboxes/*.json
  → 变化时读取，对比缓存的消息数量 → 仅推送新增消息 → new_message

~/.claude/tasks/{team}/*.json
  → 变化时读取任务内容 → 推送 task_update

定时轮询（每 5 秒）:

~/.claude/projects/{project-hash}/{session-id}.jsonl
~/.claude/projects/{project-hash}/{session-id}/subagents/*.jsonl
  → 解析新增行的 usage 字段 → 累加到各成员统计 → 推送 token_update
```

JSONL 文件采用定时轮询而非 fsnotify，原因：
- JSONL 文件写入频率很高，每条消息都追加
- fsnotify 会产生大量事件，防抖处理复杂
- 定时轮询（记录文件 offset）更稳定，5 秒间隔足够

关键处理：
- **消息去重**：缓存每个 inbox 文件的上次消息数量，只推送增量
- **防抖**：文件写入可能触发多次 fsnotify 事件，200ms 窗口内合并
- **JSON 容错**：写入过程中可能读到不完整 JSON，需 try/catch 忽略

### 6.2 四个核心面板

#### 面板一：成员状态卡片

- 每个 Agent 一张卡片
- 显示：名称、角色（agentType）、模型、运行模式（in-process/tmux）
- 状态颜色：绿色=活跃、黄色=空闲、灰色=已关闭
- 数据来源：`config.json` → `members[]`

#### 面板二：任务看板（Kanban）

- 三列布局：待分配 | 进行中 | 已完成
- 每张卡片：任务标题、owner、依赖关系
- 数据来源：`tasks/{team}/*.json`

#### 面板三：实时通信流

- 时间线形式，新消息从底部追加
- 每条消息：时间戳、发送方 → 接收方、消息内容
- 自动滚动到最新
- 数据来源：`inboxes/*.json` 的增量差异

#### 面板四：Token 消耗统计

- 顶部显示团队总消耗：总 input/output tokens、预估费用
- 每个成员一行：名称、模型、input/output tokens、预估费用、占比进度条
- Cache 命中率指标：`cache_read / (cache_read + input)` 百分比
- 数据每 5 秒自动刷新
- 数据来源：JSONL 会话记录文件的 `usage` 字段累计

## 七、数据流时序

```
Agent A 发消息给 Agent B 时的完整监控链路：

1. Agent A 调用 SendMessage(to="agent-b", message="...")
2. Claude Code 写入 ~/.claude/teams/{team}/inboxes/agent-b.json
3. fsnotify 检测到 agent-b.json 文件变化事件
4. watcher 读取文件，对比缓存的消息数量
5. 提取新增消息（index >= 缓存长度的部分）
6. 构造 WebSocket 事件 {event: "new_message", data: {...}}
7. 推送给所有连接了该 team 的前端客户端
8. 前端在通信流面板追加一条消息
9. 整个延迟 < 500ms
```

## 八、团队识别

启动时扫描 `~/.claude/teams/` 下所有子目录，读取每个 `config.json` 提取：

| 字段 | 用途 |
|------|------|
| `name` | 团队名称 |
| `leadSessionId` | 关联到具体 Claude Code 会话 |
| `members[0].cwd` | 工作目录，帮助用户区分不同项目 |
| `createdAt` | 创建时间 |
| `members.length` | 成员数量 |

前端下拉框展示这些信息，用户选择要监控的团队。
